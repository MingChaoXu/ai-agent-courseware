课题3.8: 用户记忆系统实验

从「通用工具」到「专属助手」——记忆系统让Agent记住用户偏好和历史上下文

步骤1: 三种记忆架构对比

📝
扁平笔记
[{content, timestamp}, ...] 简单易实现，但无结构化，检索效率低
🗂️
JSON层级
{category: {key: value}} 结构清晰，但层级固定，扩展性差
🧠
高级JSON卡片
[{category, key, value, timestamp, metadata}] 灵活可扩展，支持检索和上下文注入
Python - 三种架构代码对比
# 架构1: 扁平笔记 flat_memory = [] flat_memory.append({"content": "用户姓名：王主任", "timestamp": time.time()}) flat_memory.append({"content": "负责领域：社会综合治理", "timestamp": time.time()}) # 优缺点: 简单易实现 / 无结构化，检索效率低 # 架构2: JSON层级 json_memory = { "个人信息": {"姓名": "王主任", "职务": "综治中心主任"}, "业务偏好": {"简报风格": "数据先行", "关注领域": "矛盾纠纷"}, "交互记录": {"咨询次数": 5, "上次话题": "劳资纠纷处理"} } # 优缺点: 结构清晰 / 层级固定，扩展性差 # 架构3: 高级JSON卡片（MemoryManager） mm = MemoryManager(filepath="memory_demo.json") mm.add_fact("个人信息", "姓名", "王主任") mm.add_fact("个人信息", "职务", "综治中心主任") mm.add_fact("业务偏好", "简报风格", "数据先行") mm.add_fact("业务偏好", "关注领域", "矛盾纠纷") # 优缺点: 灵活可扩展 / 实现复杂度较高 # MemoryManager 核心方法： # add_fact(category, key, value, metadata) — 添加记忆 # search(query, top_k) — 关键词检索 # get_context_for_prompt(query) — 生成可注入Prompt的记忆摘要

维度无记忆有记忆
部门归属泛化（未知）XX区住建局

核心洞察：记忆系统让Agent从「通用工具」进化为「专属助手」
没有记忆的Agent，每次对话都是「新朋友」——它不知道你是谁、做什么、喜欢什么风格。
有记忆的Agent，就像「老同事」——它知道你的部门、负责的业务、偏好的文风，甚至记得上次帮你做了什么。

选择建议：轻量场景用扁平笔记，业务场景用JSON层级，复杂场景用高级JSON卡片（MemoryManager）。
记忆注入Prompt是最简单有效的上下文工程手段之一。

课题4: 金融领域自动化合同审核（含HITL审批）

Agent自动识别风险条款 + 人工审批(HITL)流程 = 自动化与安全性的平衡

步骤1: 创建合同审核智能体

合同审核5大维度
1. 利率条款：检查是否存在银行单方面调价权、浮动利率未设上限
2. 担保条款：担保方式是否明确、评估价值是否充足、保证期间是否合理
3. 违约条款：违约金比例是否过高（>3%为高风险）、交叉违约是否过于宽泛
4. 争议解决：仲裁条款是否明确仲裁地点和规则
5. 权利失衡：是否存在单方面解除权、不对等的提前还款违约金
Python
agent = client.create_agent( name="合同审核智能体", description="银行信贷合同批量审核，识别风险条款并生成审核报告，支持HITL审批", system_prompt="""你是一个专业的合同审核助手，负责审核银行信贷合同并识别风险条款。 审核维度： 1. 【利率条款】检查是否存在银行单方面调价权、浮动利率未设上限 2. 【担保条款】检查担保方式是否明确、担保物评估价值是否充足、保证期间是否合理 3. 【违约条款】检查违约金比例是否过高（>3%为高风险）、交叉违约条款是否过于宽泛 4. 【争议解决】检查仲裁条款是否明确仲裁地点和规则 5. 【权利失衡】检查是否存在单方面解除权、不对等的提前还款违约金 风险等级判定： - 🟢低风险：无风险条款或仅有轻微提示项 - 🟡中风险：存在1-2条中等风险条款 - 🔴高风险：存在3条及以上风险条款，或任一条款违约金>5% 输出格式： ## 合同审核报告 **风险等级**: 🟢低风险 / 🟡中风险 / 🔴高风险 **风险条款数**: X条 **需人工审批**: 是/否 ### 风险条款详情 | 序号 | 条款 | 风险类型 | 风险等级 | 问题描述 | 修改建议 | ### 审核建议""", model="default" )

步骤2: 合同审核测试

用两个典型案例测试：一个低风险合同、一个高风险合同。

1

低风险合同：个人住房贷款

贷款金额80万元，年利率4.2%（固定），期限30年，等额本息还款。担保方式：所购房产抵押，评估价值120万元。违约金：提前还款收取1%手续费。争议解决：XX市仲裁委员会。

预期结果：🟢低风险，自动通过

2

高风险合同：企业经营贷款

贷款金额500万元，年利率为浮动利率（银行有权单方面调整），期限5年。担保方式：企业全部资产抵押（未明确评估价值）。违约金：逾期还款收取每日0.5%罚息+5%违约金。银行有权在任何时候要求提前偿还全部贷款。争议解决：由银行总部所在地法院管辖。

预期结果：🔴高风险，需人工审批（至少4处风险条款）

步骤3: HITL (Human-in-the-Loop) 人工审批流程

Agent审核合同 → 风险分级 → 分级处理 │ ┌──────────┼──────────┐ │ │ │ 🟢低风险 🟡中风险 🔴高风险 │ │ │ 自动通过 建议人工复核 必须人工审批 │ │ │ ▼ ▼ ▼ 完成 复核通过? 3种审批选项: │ 1. 同意拒绝 y─►完成 2. 修改后通过 n─►详细复核 3. 强制通过(需原因) │ ▼ 审批留痕
HITL设计4大要点
1. 风险分级：低风险自动→中风险复核→高风险审批，避免一刀切
2. 审批留痕：所有人工决策必须记录原因，可追溯
3. Agent→Human：Agent给出建议，人做最终决策（Agent不直接执行高风险操作）
4. 异常兜底：强制通过需写明特殊原因（如「战略客户特批」），防止滥用
Python - HITL审批逻辑
for title, risk in risk_labels: if risk == "high": print(f"📋 合同「{title}」需人工审批") print(" Agent建议: 🔴高风险，建议拒绝或要求修改条款") print(" ┌─────────────────────────────────┐") print(" │ 人工审批选项: │") print(" │ 1. 同意Agent建议（拒绝合同） │") print(" │ 2. 修改后通过（指定修改要求） │") print(" │ 3. 覆盖风险标记（强制通过） │") print(" └─────────────────────────────────┘") human_decision = input(" 请选择审批操作 (1/2/3): ") if human_decision == "1": print(" ✅ 审批结果: 同意Agent建议，合同已拒绝") elif human_decision == "2": modification = input(" 请输入修改要求: ") print(f" ✅ 审批结果: 要求修改后通过") # Agent根据修改要求提出修改方案 result = client.chat(agent_id, f"根据人工审批要求，对合同「{title}」提出具体修改方案: {modification}") elif human_decision == "3": reason = input(" 请输入强制通过原因: ") print(f" ⚠️ 审批结果: 覆盖风险标记，强制通过。原因: {reason}") print(" 📝 审批记录已存档") elif risk == "medium": print(f"📋 合同「{title}」建议人工复核") else: print(f"📋 合同「{title}」🟢低风险，自动通过 ✅")

mm = MemoryManager(filepath="hr_review_memory.json") # 历史审查记录 mm.add_fact("审查记录", "2025-12-加班制度", "发现3处违规：加班费计算标准低于法定、调休时限过短、强制加班条款", metadata={"risk_level": "high", "status": "已修改"}) mm.add_fact("审查记录", "2026-01-请假制度", "整体合规，建议补充：病假工资计算方式、孕期特殊保护条款", metadata={"risk_level": "low", "status": "已补充"}) mm.add_fact("审查记录", "2026-03-薪酬制度", "发现2处风险：绩效工资占比超50%、未明确发放时间", metadata={"risk_level": "medium", "status": "修改中"}) # 合规基准（法律底线） mm.add_fact("合规基准", "加班费标准", "工作日150%/休息日200%/法定节假日300%，不得以调休替代法定节假日加班费") mm.add_fact("合规基准", "试用期上限", "合同期<1年≤1个月，1-3年≤2个月，≥3年≤6个月，同一用人单位只能约定一次")

步骤1-2: 创建记忆联动审查智能体 + 审查测试

记忆联动的价值：一致性审查
传统审查：每次审查都是独立的，无法发现新制度与旧制度的冲突。
记忆联动审查：自动关联历史审查结论——
- 加班费审查过3次了，每次都发现不同问题，这次要重点检查
- 上次加班制度的加班费标准低于法定，这次新制度是否修复了这个问题？
- 新制度的试用期条款是否与合规基准一致？

这就是「一致性审查」——不仅检查单份制度的合规性，还检查多份制度之间的一致性。
Python - 待审查的考勤制度
policy_text = """ 《XX公司考勤与加班管理制度》 第三条 工作时间 公司实行标准工时制，每日工作8小时，每周工作6天。 ← ⚠️每周6天违法！ 第五条 加班管理 5.1 公司因生产经营需要安排加班的，员工应服从安排。 ← ⚠️强制加班 5.2 加班费统一按基本工资的100%计算，不分工作日/休息日/法定节假日。 ← ⚠️低于法定！ 第七条 迟到处罚 迟到15分钟以内扣款50元，15-30分钟扣款100元，30分钟以上按旷工处理。 第九条 试用期 管理岗位试用期6个月，普通岗位试用期3个月。 ← ⚠️普通岗位3个月超限 试用期工资为正式工资的70%。 ← ⚠️低于80%法定下限 """ result = client.chat(agent_id, f"请审查以下公司制度，识别所有不合规条款，并对比历史审查记录检查一致性：\n{policy_text}")

课题7.5: Prompt消融实验

科学评估Prompt各组件的贡献度——用数据说话，而非凭感觉

消融实验方法论

消融实验（Ablation Study）原理
消融实验是Prompt工程的核心评估方法：
原理：逐个移除Prompt组件，观察输出质量变化，量化各组件贡献度

典型Prompt组件：
1. System Prompt（系统提示）← 角色定义、行为约束
2. Few-shot Examples（少样本示例）← 输出格式参照
3. Tool Descriptions（工具描述）← 工具选择依据
4. Reasoning Chain（推理链引导）← 复杂推理能力
5. Tone/Style（语气风格）← 输出语气控制

实验步骤：
1. 建立Baseline（完整Prompt配置）
2. 逐个移除组件，记录输出变化
3. 用评估函数量化差异
4. 排序各组件贡献度

合同审核场景消融实践

用同一个高风险合同测试4种Prompt配置：

配置说明预期影响
baseline（完整配置）角色+审核维度+输出格式基线，得分最高
no_system_prompt仅「请审核以下合同」角色和约束全丢，影响最大
no_tool_description无审核维度指引审核零散无体系
informal_tone改为非正式语气输出不专业但核心信息仍在
Python - 消融实验核心代码
prompt_configs = { "baseline": { "system_prompt": """你是一个专业的金融合同审核助手。 审核维度： 1. 利率条款风险 2. 担保条款风险 3. 违约条款风险 4. 权利平衡性 输出格式：风险等级 | 风险条款数 | 详细审核意见 | 修改建议""", }, "no_system_prompt": { "system_prompt": "请审核以下合同。", }, "no_tool_description": { "system_prompt": """你是一个合同审核助手。 请审核合同。（无审核维度指引）""", }, "informal_tone": { "system_prompt": """嘿！来帮我看看这个合同有没有坑哈～ 大概看看这几点就行：- 利率有问题不 - 担保靠谱不 - 违约金离谱不 随便说说就好，不用太正式～""", }, } # 评估函数：量化审核质量 def evaluate_contract_review(question, answer): score = 0.0 if not answer or len(answer) < 20: return 0.0 risk_keywords = ["风险", "违约", "担保", "利率", "霸王", "不合规", "违规", "建议修改"] found = sum(1 for kw in risk_keywords if kw in answer) score += found / len(risk_keywords) * 0.6 structure_keywords = ["|", "1.", "2.", "报告", "审核"] found_struct = sum(1 for kw in structure_keywords if kw in answer) score += found_struct / len(structure_keywords) * 0.2 score += min(0.2, len(answer) / 1000) return round(min(1.0, score), 3)
消融实验核心洞察
- 系统提示贡献最大 → 定义了角色、审核维度、输出格式，是Prompt的「灵魂」
- 审核维度指引影响显著 → 没有指引则审核零散无体系，遗漏关键风险
- 语气风格影响中等 → 非正式语气导致输出不专业，但核心信息通常仍在
- Baseline配置综合最优 → 各组件协同效果 > 单组件之和

实践建议：做Prompt优化时，先用消融实验定位最关键组件，系统提示是最值得投入优化时间的部分。
