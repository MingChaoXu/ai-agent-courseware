AI发展与大模型时代

从规则引擎到涌现能力，理解AI为什么在今天真正可用 · 20分钟

从规则到涌现：AI的三次范式跃迁

1950s-1980s
符号主义：规则引擎
人类编写IF-THEN规则，机器执行。专家系统如MYCIN可诊断血液感染，但规则爆炸导致无法扩展。
2010s
统计学习：深度学习
从数据中自动学习特征。CNN识别图像、RNN处理序列，但每个任务需独立训练模型，泛化能力弱。
2020s
大语言模型：涌现能力
GPT系列模型在千亿参数规模下涌现出推理、翻译、编程等未被显式训练的能力——"量变引起质变"。
⚡ 核心概念：涌现能力（Emergent Abilities）
当模型参数量从10亿增长到1000亿时，某些能力不是渐进提升的，而是在某个阈值突然"涌现"出来。这就像水温从99°C到100°C——不是"更热的水"，而是"质的改变"。

涌现的关键能力包括：
1. 链式推理（Chain-of-Thought）：能分步骤解决复杂问题
2. 指令遵循（Instruction Following）：理解并执行自然语言指令
3. 上下文学习（In-Context Learning）：无需微调，从示例中学习新任务

这三个能力是Agent技术得以实现的前提条件——没有推理能力就无法规划，没有指令遵循就无法控制，没有上下文学习就无法适配新场景。

大模型的能力边界：能做什么，不能做什么

能力维度LLM 擅长 ✅LLM 不擅长 ❌
语言理解意图识别、情感分析、文本摘要精确数学计算、逻辑证明
知识问答通识领域、常见专业领域实时信息、超长尾知识
内容生成文案、代码草稿、报告框架需要精确数据的报表
推理规划多步骤任务拆解、方案设计需要外部验证的推理链
工具调用选择合适工具、构造参数执行实际操作、访问实时数据
⚠ 关键风险：幻觉（Hallucination）
LLM会自信地生成看似合理但实际错误的内容。这是因为它不是在"检索事实"，而是在"预测下一个Token"。当训练数据中缺少相关信息时，模型会"编造"听起来合理的答案。

应对策略：
1. RAG：先检索真实文档，再基于文档生成回答
2. 工具调用：让Agent查数据库、调API，而非凭记忆回答
3. 人机协同（HITL）：高风险决策必须经人工确认
4. System Prompt约束：明确告知"不知道就说不知道"

LLM核心原理

理解大语言模型的运作机制，建立正确的技术直觉 · 30分钟

Transformer架构：一切的基础

⚡ Transformer的核心创新：自注意力机制
2017年Google论文《Attention Is All You Need》提出Transformer，核心思想：

每个词都"看"到句子中的所有其他词，通过计算注意力权重决定"谁对我最重要"。

示例：句子「银行利率调整影响银行股价」
- 第一个「银行」→ 关注「利率」(权重0.7) + 「调整」(权重0.2)
- 第二个「银行」→ 关注「股价」(权重0.6) + 「调整」(权重0.3)

同样的"银行"，因为上下文不同，关注点完全不同——这就是自注意力的威力。

Transformer vs RNN：
- RNN：从左到右逐词处理，长距离信息衰减 → 无法并行
- Transformer：所有位置同时计算注意力 → 可并行 + 长距离建模
Transformer 编码器结构（简化）： 输入文本 → Token嵌入 + 位置编码 │ ▼ ┌─── 多头自注意力 ───┐ │ Q·K^T / √d → Softmax │ ← 计算每个Token对其他Token的关注度 │ 结果 × V │ ← 加权汇总信息 └────────┬──────────┘ │ + 残差连接 & 层归一化 ▼ ┌─── 前馈神经网络 ───┐ │ Linear → ReLU → Linear │ ← 逐位置非线性变换 └────────┬──────────┘ │ + 残差连接 & 层归一化 ▼ 编码输出 GPT系列 = Transformer解码器堆叠（仅保留自注意力+前馈层） BERT系列 = Transformer编码器堆叠
原始Transformer架构：左侧编码器栈 + 右侧解码器栈，用于机器翻译等序列到序列任务。来源：Jay Alammar
三大家族对比：GPT-2 仅用解码器栈（自回归）、BERT 仅用编码器栈（双向）、Transformer-XL 解码器+片段递归。来源：Jay Alammar
自注意力 vs 掩码自注意力：左图普通自注意力（BERT）可以看到所有位置；右图掩码自注意力（GPT）只能看到当前及之前的Token，防止"偷看"未来。这是 GPT 自回归生成的关键机制。来源：Jay Alammar

注意力机制与自回归生成

💡 核心原理：LLM是"逐Token预测"的
GPT系列模型的工作方式是自回归（Autoregressive）：

输入：「公积金提取需要」
Step 1: 预测下一个Token → 「哪些」
Step 2: 输入变为「公积金提取需要哪些」→ 预测 → 「材料」
Step 3: 输入变为「公积金提取需要哪些材料」→ 预测 → 「？」
...

每次只预测一个Token，然后把预测结果拼到输入中，再预测下一个。这就是为什么：
1. 生成越长越慢（每步都要重新计算注意力）
2. 早期的错误会"滚雪球"式传播（幻觉的根源之一）
3. max_tokens参数控制最多生成多少Token
GPT-2 逐Token输出：模型每次只生成一个Token，生成后立刻拼入输入序列，作为下一步的输入。这就是"自回归"的核心过程。来源：Jay Alammar
Step 1：输入起始Token <s>，经过所有解码器层后，对词表打分，选出概率最高的Token（如"the"）。
Step 2：将上一步的输出拼入输入，新的Token路径被激活，各层复用已有Key/Value，预测下一个Token。
自回归循环：每一步的输出Token都会"反馈"到输入端，形成"输入→预测→拼入→再预测"的循环，直到生成结束符或达到 max_tokens 上限。这也解释了为什么生成速度与输出长度成正比。来源：Jay Alammar

Token化与上下文窗口

🔧 Token：LLM的最小处理单位
文本不是按"字"或"词"输入模型的，而是按Token。Token是分词器（Tokenizer）划分的最小单元：

英文："Hello world" → [Hello, world] = 2 tokens
中文："你好世界" → [你好, 世界] ≈ 2-4 tokens
数字："3.14159" → [3, ., 14159] = 3 tokens

关键规则：
1. 1个汉字 ≈ 1-2个Token，1个英文单词 ≈ 1-2个Token
2. 1个Token ≈ 0.75个英文单词 ≈ 0.5个中文字（粗略估算）
3. 上下文窗口 = 模型一次能处理的最大Token数（如4K/8K/128K）
4. Token数直接影响成本和延迟
模型上下文窗口约等于中文字数典型用途
GPT-3.54K tokens~2,000字简单对话
GPT-48K-128K tokens~4,000-64,000字长文档处理
Claude 3200K tokens~100,000字超长文档分析
国产大模型8K-128K tokens~4,000-64,000字政务/企业场景
1张图片（512×512）~258 tokens≈ 130字图像理解
1张图片（1920×1080）~2,042 tokens≈ 1,000字高清图片分析
📸 图像↔Token：视觉信息的数字化代价
多模态模型将图像也转换为Token，但一幅图的Token消耗远超想象：

图像Token化过程：
1. 将图片切成 14×14像素的小块（Patch）
2. 每个Patch经过PatchEmbed变为一个向量 → 2×2 PatchMerger合并 → 每~32×32像素 ≈ 1个Token
3. 再加2个特殊Token（起止标记）

Token计算公式：num_tokens = (H'/32) × (W'/32) + 2（H'/W'为缩放后的分辨率）

实际示例：
• 512×512 图片 → (16×16)+2 = 258 tokens ≈ 130个中文字
• 1920×1080 图片 → (~44×~46)+2 ≈ 2,042 tokens ≈ 1,000个中文字
• 4K图片(3840×2160) → ~8,194 tokens ≈ 4,000个中文字

实战启示：发送10张截图给大模型 ≈ 消耗1-2万Token，等于喂了一整篇论文！图片越大、分辨率越高，Token消耗指数级增长。
⚠ 上下文窗口 ≠ 有效注意力
虽然模型技术上支持128K上下文，但并非所有128K Token都能被同等关注。研究表明：
1. 模型对开头和结尾的内容注意力最强（"首因效应"+"近因效应"）
2. 中间部分的内容容易被"遗忘"（Lost in the Middle现象）
3. 这就是为什么System Prompt放在开头（首因效应）、最新对话放在末尾（近因效应）
4. 上下文压缩策略（Day 3学习）的核心目的就是解决窗口受限问题

温度与采样策略

🔧 Temperature：控制输出的"创造性"
temperature参数控制模型从概率分布中采样的"随机性"：

temperature = 0（贪心解码）：每次选概率最高的Token → 输出确定性强、无创意
temperature = 0.3（低温度）：大部分时候选高概率Token，偶尔有变化 → 适合事实性问答
temperature = 0.7（中温度）：较平衡 → 适合一般对话
temperature = 1.0（高温度）：更多探索低概率Token → 适合创意写作

实战建议：
- 政务问答/合同审核：temperature=0.1~0.3（准确性优先）
- 文案生成/营销策略：temperature=0.6~0.8（创意性优先）
- 代码生成：temperature=0.2（确定性优先）
🔧 Top-K 与 Top-P：精细控制采样空间
Temperature控制概率分布的"陡峭程度"，而Top-K和Top-P控制从哪些Token中采样——两者配合使用效果更精细：

Top-K Sampling：
只从概率最高的K个Token中采样，其余直接截断。
• top_k=1：等价于贪心解码（只选最高的1个）
• top_k=50：从概率Top 50中按权重随机选（常用默认值）
• K越大→多样性越强，K越小→确定性越强

Top-P（Nucleus Sampling）：
从概率最高的Token开始累加，直到累计概率 ≥ P，只在这个最小集合中采样。
• top_p=0.1：极窄，几乎只选最高概率Token
• top_p=0.9：常用默认值，累计覆盖90%概率质量
• 优势：自适应——常见词自动少选（概率集中），罕见词自动多选（概率分散）

三者协同规则：先Top-K截断 → 再Top-P截断 → 最后Temperature缩放 → 采样
API中通常同时设置，更严格的那个生效。实战建议：
- 确定性场景：temperature=0（此时Top-K/P不生效）
- 平衡场景：temperature=0.7, top_p=0.9（Top-P为主，Temperature微调）
- 创意场景：temperature=1.0, top_k=0, top_p=0.95（放开Top-K限制）
⚙ vLLM 部署核心参数
vLLM是当前最流行的开源推理引擎，基于PagedAttention技术实现高效KV Cache管理。以下是vllm serve启动时最关键的参数：

一、模型与加载
• --model：模型路径或HuggingFace ID（如Qwen/Qwen2.5-72B-Instruct）
• --tensor-parallel-size（-tp）：张量并行度，模型分层放到多张GPU上。72B模型需tp=2（2×A100），7B/14B单卡即可tp=1
• --pipeline-parallel-size（-pp）：流水线并行度，按层切分到不同GPU。通常tp优先，pp在跨节点时使用
• --quantization：量化方式。awq（4bit，A100/WIDER适用）/ gptq（4bit）/ fp8（H100原生支持）。量化后显存降50-75%，吞吐提升但精度略损

二、KV Cache 与显存
• --gpu-memory-utilization：GPU显存利用率，默认0.9（90%给KV Cache，10%给模型权重和安全余量）。调高→更多并发，调低→更稳不易OOM
• --max-model-len：模型支持的最大序列长度。设小→KV Cache占用少→并发更高。如模型支持128K但你只需要4K，设4096可大幅提升并发
• --block-size：PagedAttention的内存块大小（默认16 Token），一般不需调整

三、调度与并发
• --max-num-seqs：单批次最大并发序列数，默认256。受显存约束，实际并发=可用KV Cache ÷ 平均序列长度
• --max-num-batched-tokens：单批最大Token总数（prefill阶段），默认自动推算。越大吞吐越高，但prefill延迟也增
• --enable-chunked-prefill：分块预填充，将长prompt的prefill拆成小块与decode交替执行，避免长请求阻塞短请求

四、服务与输出
• --host / --port：监听地址，默认0.0.0.0:8000，兼容OpenAI API格式
• --served-model-name：对外暴露的模型名（API中model字段），可与实际模型名不同，方便迁移切换
• --enable-prefix-caching：前缀缓存（APC），相同System Prompt的请求复用KV Cache，多轮对话/批量场景提速显著
• --disable-log-requests：关闭请求日志（生产环境建议关闭，减少IO开销）

⚠ vLLM 部署常见陷阱
1. max-model-len 设太大：128K窗口下KV Cache占用极大，并发暴跌。如果业务只需4K，设4096可将并发提升10倍以上。
2. gpu-memory-utilization 拉满到0.99：看似用满显存，但模型加载、临时Tensor、框架开销都需要余量，0.95以上容易OOM崩溃。建议0.85-0.9。
3. 张量并行≠线性加速：tp=2不是单卡2倍吞吐，实际约1.5-1.8倍（通信开销）。跨节点tp更慢，此时应考虑pp流水线并行。
4. 量化后不校验精度：AWQ/GPTQ在数学推理、长文本场景精度损失明显。部署后必须用业务数据集做回归评测，不能只看benchmark。

Agent = LLM + 上下文 + 工具

理解Agent的核心公式和三大组件 · 30分钟

Agent核心架构：LLM（推理）+ Context（上下文）+ Tools（工具）通过Thought→Action→Observation循环逐步逼近答案

Agent核心公式

Agent 核心公式： Agent = LLM + Context + Tools ┌─────────────────────────────────────────┐ │ Agent 运行循环 │ │ │ │ ┌─────┐ ┌─────────┐ ┌───────┐ │ │ │ LLM │ ◄─┤ Context │ ◄─┤ Tools │ │ │ │(推理)│──►│(上下文) │──►│(工具) │ │ │ └─────┘ └─────────┘ └───────┘ │ │ │ ▲ │ │ │ │ │ │ │ │ ▼ │ ▼ │ │ Thought ──► Action ──► Observation │ │ │ │ │ │ └─────► 更新Context ◄────────────┘ │ │ │ │ 三大组件： │ │ 1) LLM：大语言模型，负责推理和决策 │ │ 2) Context：对话历史+系统提示+工具结果 │ │ 3) Tools：可调用的外部工具集 │ └─────────────────────────────────────────┘ ReAct (Reasoning + Acting) 循环： 每轮迭代 = Thought → Action → Observation 循环直到得到最终答案或达到最大迭代次数
⚡ 三大组件缺一不可
没有LLM：无法推理，Agent无智能 → 退化成规则脚本
没有Context：无法记忆，每轮都是"失忆"状态 → 无法做多轮对话
没有Tools：无法行动，只能纯文本对话 → 退化为ChatBot

三者组合才能产生真正的Agent行为：LLM推理需要Context，Context由Tool结果填充，Tool选择由LLM决策。

三大组件详解

组件1：LLM — 推理引擎

LLM在Agent中的角色不是"聊天"，而是推理和决策。它需要决定：

  • 当前情况是否需要调用工具？调用哪个？
  • 工具返回的结果是否足够？是否需要继续？
  • 如何将多个工具的结果综合成最终答案？

组件2：Context — 工作记忆

Context是Agent的"工作记忆"，包含三个层次：

层次内容类比
System Prompt角色定义、行为约束、输出格式岗位说明书
对话历史用户与Agent的交互记录会议纪要
工具结果工具调用返回的信息调查报告

组件3：Tools — 行动能力

工具让Agent从"纸上谈兵"变为"能办实事"。三大工具类型：

📡
感知工具
获取外部信息：网络搜索、多模态理解、文件系统、数据库查询、公共数据源
⚙
执行工具
执行操作：代码解释器、文件操作、系统命令、图表生成（安全沙箱保护）
🤝
协作工具
人机协同：浏览器自动化、通知推送、人工审批、定时任务

从ChatBot到Agent的演进

CMU/Yale/Amazon联合发表的综述论文《Agent Harness Engineering: A Survey》提出绑定约束论（Binding-Constraint Thesis）：

长程任务的基准方差由执行Harness驱动，而非模型本身。

Harness Engineering：Agent的"变速箱+刹车"

🚗 类比：Agent是一辆车
模型 = 引擎（马力大，给油就响）
Prompt = 方向盘（你打方向引擎带着你走）
交互程序 = 车轮（承载它上路）

但引擎+方向盘+车轮本身不是车——你不能开着引擎上路。
你还需要：变速箱（Harness编排）、制动器（Harness约束）、仪表盘（Harness可观测性）、刹车（Harness治理）。

公式：Agent = LLM（大脑）+ Harness（全部约束与支撑体系）

场景：运维Agent自动处理服务器告警工单

Trigger：监控系统推送告警事件（CPU > 95%持续5分钟）
Agent Loop 第1轮：Agent推理 → 调用ssh_tool查看Top进程 → 发现是批处理任务占用
Validator：检查Agent输出是否包含"进程名"和"CPU占比" → 通过
Agent Loop 第2轮：Agent推理 → 调用query_history查询该任务历史 → 发现是每月例行任务
Validator：检查是否引用了真实历史记录 → 通过
Agent Loop 第3轮：Agent生成处置建议 → "建议将批处理任务调度至凌晨2点执行"
Validator：建议中包含具体cron表达式 + 影响范围评估 → 通过 ✓
Guardrails：本轮消耗Token 1,847 < 上限2,000；耗时23秒 < 60秒超时 → 继续
Done：输出处置工单，自动派发给运维人员确认

如果不通过会怎样？ 第2轮Agent未引用历史记录 → Validator判定不通过 → Feedback Loop自动补充"请查询该任务最近3次执行记录" → 重新进入Agent Loop

应用示例：政务文档审查Loop

场景：合同审查Agent自动标注风险条款

Trigger：用户上传合同PDF → 系统提取文本后触发
Agent Loop 第1轮：Agent逐条款审查 → 标注3处潜在风险
Validator：检查每处标注是否包含条款原文+风险原因+修改建议 → 2处通过，1处缺少修改建议
Feedback Loop：自动补充"请为第3处风险补充修改建议" → Agent补充建议 → 再次验证 → 全部通过 ✓
Memory：记住本合同的行业类型和甲方信息，后续条款审查参考
Guardrails：审查条款超过50条时，自动拆分为多轮处理，单轮不超过30条；累计Token超5,000时强制输出已审查结果
Done：输出标注完的风险审查报告

Harness vs Loop：什么关系？

🏠 Harness是地基，Loop是上层建筑
Harness 解决"Agent能不能跑起来"——造路、设红绿灯、装摄像头（ETCLOVG静态基础设施）
Loop 解决"Agent能不能自己跑完"——设计公交线路的调度系统（触发-执行-评估-修正的动态循环）

两者的区别不在于组件不同（都会涉及工具、验证、上下文），而在于关注重心不同：
- Harness → 静态约束：环境怎么搭、工具怎么挂、安全怎么控
- Loop → 动态编排：循环怎么转、错误怎么纠、结果怎么验

在真实生产Agent中，两者是互补共生的。没有Harness的Loop是裸奔（没有安全网），没有Loop的Harness是摆设（跑不起来）。
⚠ Loop Engineering 的三大风险
1. 成本失控：自动循环可能无限重试——没有Guardrails的Loop = 没有刹车的车，Token账单可能从$0.5跑到$50
2. 方向漂移：Validator太弱时，Agent会在错误方向上越走越远——第1轮改错文件，第47轮还在改错文件
3. 并非万能：探索性/创意性任务（头脑风暴、战略规划），Loop比手动Prompt更差——需要人的判断力，不是循环次数

OpenClaw：文件系统记忆与Skill生态

📦 长记忆机制：Markdown文件就是数据库
OpenClaw最独特的设计是不使用数据库，而是用普通Markdown文件存储记忆。每次对话开始时，Agent会先读取workspace目录下的文件，根据文件内容决定如何回复。这种设计的好处是：可读、可编辑、可Git版本控制——用户可以直接打开文件查看和修改Agent的"记忆"。

核心文件体系：
文件含义存放内容类比
SOUL.md灵魂文件Agent的性格、语气、价值观、行为风格一个人的性格底色
USER.md用户画像用户偏好、工作习惯、沟通风格"你老板是什么样的人"
AGENTS.md工作规范怎么干活、安全边界、操作约束岗位说明书
MEMORY.md长期记忆重要事项、项目上下文、关键决策笔记本
daily-log/日期.md每日日志当天关键事件和对话摘要日记本
HEARTBEAT.md心跳任务定期自动执行的任务清单闹钟/定时器
DREAMS.md梦境整合后台记忆整理的中间结果（类似睡眠整合记忆）做梦时整理白天的记忆
IDENTITY.md身份设定Agent的名字、角色定位名片
TOOLS.md工具配置可用工具列表及参数配置工具箱清单
🔧 Skill系统：写文档比写代码快
OpenClaw的Skill不是代码插件，而是Markdown格式的指令文档。每个Skill是一个.md文件，告诉Agent"遇到某类任务该怎么做"。

工作流程：Agent启动时只加载Skill的元数据（名称+描述+触发词，约97字符/个），不加载全文。当Agent判断需要某个Skill时，先调用read_skill读取完整内容，再按指令执行——这就是"Read Before Acting"模式，避免一次性加载所有Skill浪费Token。

三级优先级：workspace级（最高）> 用户级（~/.openclaw/skills）> 内置级。同名Skill高优先级覆盖低优先级，支持热更新。

与Claude Code的对比：Claude Code用代码定义工具（ACI协议），OpenClaw用文档定义技能。前者精确但开发成本高，后者灵活但依赖LLM理解能力。
💬 IM集成：把Agent拉进微信群
OpenClaw原生支持接入微信、企业微信、钉钉、飞书、QQ等主流IM平台。用户在IM中@Agent或发私聊，即可触发Agent执行任务——不需要打开额外的App。

9级路由优先级：从"特定用户+特定群组+消息类型"（级9，最高）到"全局默认Agent"（级1），路由是确定性规则匹配（延迟<1ms），不消耗LLM调用。

NO_REPLY机制：Agent判断不需要回复时返回NO_REPLY。在群聊中不是每条消息都需要Agent回复，给AI"沉默权"是一个重要设计——避免Agent刷屏。

Heartbeat主动巡检：默认约30分钟自动触发一次，检查有无待处理事件。无事件时静默退出（HEARTBEAT_OK），有事件时主动在会话中执行。配合Cron定时调度，实现"Agent主动找你"而非"你找Agent"。

Hermes：能"自我进化"的Agent

🧠 闭环学习：唯一能从使用中自主进化的Agent
Hermes由Nous Research开发（2026年2月发布，6周内GitHub 7万星），核心理念是"与用户共同成长的数字同事"。与其他Agent最大的区别是闭环学习——它能自动从你的使用过程中发现模式，生成新技能：

Stage 1 任务执行：Agent正常完成你的任务
Stage 2 Nudge Engine：后台扫描对话，识别"这个操作模式反复出现，值得沉淀为技能"
Stage 3 技能生成：自动fork一个子Agent，撰写SKILL.md技能文档
Stage 4 技能修正：通过RL强化学习+人工审阅，确保技能质量

对比：OpenClaw的Skill依赖社区下载或人工编写（被动获取），Hermes的闭环学习是主动进化（自己发现模式，自己写技能）。更多使用 → 更多技能 → 更强服务——从线性增长到指数增长。
📚 五层记忆系统
Hermes的记忆比OpenClaw更复杂，分为五层，从即时到持久递进：

L1 Transformer上下文：当前对话窗口内的即时记忆，会话结束即消失
L2 SKILL.md技能知识：沉淀的"怎么做"技能，永久保存，类似OpenClaw的Skill
L3 向量存储索引：跨会话语义检索，能"模糊回忆"相关历史信息
L4 Honcho辩证式用户建模：Hermes的独家创新——不简单记住"用户说了什么"，而是通过正-反-合辩证推理理解"用户为什么这么说"，构建对用户的深层认知模型
L5 FTS5全文检索+LLM摘要：关键词检索+语义检索双通道，像人的"联想记忆"
💡 其他核心特性
200+模型一键切换：支持OpenAI/Claude/Gemini/Qwen/DeepSeek等200+模型，不绑定单一厂商
15+消息平台接入：微信/钉钉/飞书/Telegram/Discord等，与OpenClaw的IM集成类似
agentskills.io标准：Hermes推动的技能跨框架可移植标准，Skill可以在不同Agent平台间共享
子Agent委派：主Agent可将子任务委派给子Agent并行处理，但有严格安全约束（禁止递归、禁止写共享记忆、最多2层深度）

QwenPaw：阿里千问生态的桌面Agent

🐾 从CoPaw到QwenPaw：开源桌面Agent
QwenPaw（千问爪子）是阿里巴巴开源的桌面Agent工具，前身叫CoPaw，2026年4月更名后纳入通义千问开源生态。与OpenClaw的定位差异：

OpenClaw：服务端部署为主，Gateway长驻进程，通过IM/浏览器交互，适合团队和多用户场景
QwenPaw：桌面本地部署为主，直接操作本地文件和应用，适合个人办公场景

二者也有共同点：都基于Skill文档驱动（Markdown定义技能）、都支持IM接入、都使用文件系统存储记忆。
🔧 核心功能
千问生态深度整合：默认接入Qwen系列模型，也可切换其他开源模型（DeepSeek/GLM等），支持本地Ollama部署
本地+云端双模式：敏感数据用本地模型处理，复杂任务切云端大模型，兼顾隐私和能力
全平台覆盖：macOS + Windows + Linux，与OpenClaw的跨平台能力一致
多智能体协作：正在探索多Agent协作模式，类似Hermes的子Agent委派但更轻量
开源社区驱动：MIT协议，社区贡献Skill和工具插件

💻 QoderWork：三模式分级自主性
阿里推出的桌面智能体工作台，核心设计是三种工作模式平衡自主性与控制感：
• Ask模式：只读问答，不修改任何文件——适合咨询类场景
• Agent模式：多轮迭代执行，读写操作需用户确认——适合日常办公
• Quest模式：一次性委托，全权限端到端交付——适合"下班前交代、第二天取成果"

意识系统（2026年6月上线）：短期记忆→长期记忆自动晋升，每日日志24小时后自动蒸馏为MEMORY.md，防止记忆膨胀。这与Hermes的闭环学习异曲同工，但更偏工程化。
📱 WorkBuddy：IM远程指挥PC
腾讯推出的全场景职场AI工作台，月活2000万。最大差异化是跨端远程调度（WorkBuddy Claw）：打通微信、企微、QQ、飞书、钉钉5大IM，用户在手机上发一句话，就能远程唤醒PC端Agent执行任务。

System Prompt设计：Agent的"宪法"

🔧 System Prompt三元组：角色 + 职责 + 格式
一个好的System Prompt必须包含三个要素：

1. 角色声明：告诉LLM"你是谁"
你是政务大厅前台问答Agent。

2. 职责定义：划清"该做什么"和"不该做什么"的边界
职责：理解市民诉求，判断业务类型并提取关键信息。
业务类型：户籍办理 | 社保业务 | 公积金业务 | 不动产登记 | 婚姻登记 | 其他

3. 输出格式：指定结构化输出，确保下游可解析
输出格式：JSON {"intent": "业务类型", "keywords": ["关键词"], "urgency": "普通/加急"}

为什么输出格式如此重要？因为工作流中Agent的输出是下一个Agent的输入。如果问答Agent输出自然语言，推荐Agent就无法精准提取业务类型。
System Prompt 示例
# ❌ 不好的System Prompt 你是政务助手。 # ✅ 好的System Prompt（三元组完整） 你是政务大厅前台问答Agent。 职责：理解市民诉求，判断业务类型并提取关键信息。 业务类型：户籍办理 | 社保业务 | 公积金业务 | 不动产登记 | 婚姻登记 | 其他 输出格式：JSON {"intent": "业务类型", "keywords": ["关键词"], "urgency": "普通/加急"} 示例： 用户：「我想把户口从老家迁过来」 输出：{"intent": "户籍办理", "keywords": ["户口迁移", "迁入"], "urgency": "普通"} # ✅ 更好的System Prompt（增加边界约束） 你是政务大厅前台问答Agent。 职责：理解市民诉求，判断业务类型并提取关键信息。 业务类型：户籍办理 | 社保业务 | 公积金业务 | 不动产登记 | 婚姻登记 | 其他 输出格式：JSON {"intent": "业务类型", "keywords": ["关键词"], "urgency": "普通/加急"} 约束： 1. 如果用户意图不明确，请追问而非猜测 2. 如果超出业务范围，明确告知并建议咨询12345热线 3. 不要编造政策信息 示例： 用户：「我想把户口从老家迁过来」 输出：{"intent": "户籍办理", "keywords": ["户口迁移", "迁入"], "urgency": "普通"}

Few-shot示例与输出格式控制

💡 Few-shot：用示例代替规则
LLM擅长"从示例中学习"（In-Context Learning）。与其用大量文字描述"应该怎么做"，不如给出1-3个具体示例。

Zero-shot（无示例）："请判断以下文本的情感" → 模型可能输出格式不一致
One-shot（1个示例）："示例：输入'太好了'→输出'正面'。请判断：'还行吧'" → 格式更一致
Few-shot（2-3个示例）：给出正例+反例 → 准确率和格式一致性最高

在Agent的system_prompt中，示例应该覆盖典型case + 边界case。
例如政务问答Agent的示例应包含：普通case（公积金提取）+ 边界case（超出业务范围）

常见Prompt陷阱与规避

陷阱表现规避方法
指令模糊"帮我处理一下" → Agent不知道做什么明确任务目标、输入、输出
角色缺失没有"你是XX"声明 → 回答泛泛始终声明角色+职责边界
格式未指定输出是自然语言段落 → 下游无法解析指定JSON/Markdown/编号列表
缺少约束Agent编造政策或越权回答加"不知道就说不知道"约束

语气不当政务场景用口语 → 用户不信任指定"专业正式语气"
✅ Prompt优化黄金法则
1. 角色+职责+格式三元组必须完整
2. 示例比描述更有效（Show, don't tell）
3. 关键信息放在Prompt的开头和结尾
4. 明确约束比隐式期望更可靠

ReAct循环原理

Thought-Action-Observation：Agent的核心运行机制 · 30分钟

Thought → Action → Observation 循环

⚡ ReAct = Reasoning + Acting
ReAct是Agent最核心的运行模式，由Yao et al. 2022年提出。核心思想：

Thought（思考）：LLM分析当前情况，决定下一步做什么
Action（行动）：调用工具获取信息或执行操作
Observation（观察）：接收工具返回的结果

循环：Thought → Action → Observation → Thought → Action → ... → 最终答案

为什么ReAct比纯推理或纯行动更好？
- 纯推理（Chain-of-Thought）：没有工具，无法获取新信息，容易幻觉
- 纯行动（Act-only）：没有推理，工具调用无规划，效率低且容易走错
- ReAct：推理指导行动，行动验证推理，两者协同效果最佳

完整循环演示

用户问题：「我是外地户口在苏州工作8年，想把户口迁过来，社保能一起转吗？」 ─── 迭代 1 ─── 💭 Thought: 用户问的是户口迁移的条件，我需要先搜索苏州户口迁移政策 🔧 Action: 搜索("苏州户口迁移") 👁 Observation: 苏州市户口迁移需满足：1)合法稳定住所；2)参加社保满3年；3)无犯罪记录。 办理地点：辖区派出所户籍窗口。 ─── 迭代 2 ─── 💭 Thought: 搜索结果提到需要社保满3年，我来算一下用户2018年开始交社保到现在够不够3年 🔧 Action: 计算器("2026 - 2018") 👁 Observation: 计算结果: 8 ─── 迭代 3 ─── 💭 Thought: 已交社保8年，满足3年条件。再搜索一下社保转移能否同步办理 🔧 Action: 搜索("苏州社保转移") 👁 Observation: 苏州社保转移流程：1)原参保地开具参保凭证；2)新参保地提交转移 申请；3)45个工作日内完成转移。线上可通过国家社会保险公共服务平台办理。 ─── 迭代 4 ─── 💭 Thought: 现在信息足够了，可以综合回答 ✅ 最终答案: 根据查询结果：1)户口迁移需合法稳定住所+社保满3年+无犯罪记录 （您已交8年社保，满足条件）；2)社保转移可同步在线办理， 通过国家社会保险公共服务平台，45个工作日完成。

这个4轮循环展示了ReAct的核心特征：先思考再行动，根据观察调整策略。第2轮的"计算器"调用是基于第1轮观察结果的推理决策——这正是纯推理无法做到的。

🔬 消融实验：移除组件，观察退化
消融实验（Ablation Study）是验证Agent各组件价值的方法：逐个移除上下文中的不同部分，观察Agent能力如何退化。

实验配置与结果：
消融模式Token占比效果
FULL（完整上下文）100%基线性能，完整能力
NO_HISTORY（移除对话历史）60%无法追问，多轮对话能力丧失
NO_REASONING（移除推理链）75%复杂推理断裂，准确率下降
NO_TOOL_RESULTS（移除工具返回）70%相当于无工具，回到纯聊天
MINIMAL（仅系统提示+问题）20%几乎无Agent能力

Context是Agent的"工作记忆"，每个组件都不可省略。这就是为什么后续Day 3要学上下文压缩策略——不是删减组件，而是在保留关键信息的前提下减少Token消耗。

💡 Chunking分块策略与Late Chunking
经验值（用BGE-M3）：FAQ→200-400 Token，长文档→500-800，代码→800-1500。重叠取chunk的10-20%。

Late Chunking（Jina AI 2024）：传统是「先切再嵌入」，每个chunk独立编码丢失上下文。Late Chunking是「先嵌入全文，再切分Token级向量」——每个chunk的向量保留了全文上下文。长文档场景召回率提升15-30%。
★ BGE-M3：2026年RAG事实标准
BGE-M3原生支持三种检索模式，一个模型搞定混合检索：
1. Dense Retrieval（稠密向量）→ 语义相似度
2. Sparse Retrieval（稀疏向量）→ 关键词匹配（类似BM25）
3. Multi-vector（多向量）→ ColBERT式精排

选型决策：中文为主→bge-large-zh-v1.5；多语言+开源→BGE-M3；商业最强→OpenAI text-embedding-3-large；长文档→Voyage-3

步骤2: 嵌入向量与语义检索——RAG的底层原理

⚡ 嵌入（Embedding）：文本→向量，语义相近→向量相近
嵌入是RAG的基石——将文本转化为高维向量，使语义相近的文本在向量空间中距离更近。

「公积金提取」 → [0.12, -0.34, 0.56, ...]
「住房公积金取出」→ [0.11, -0.32, 0.55, ...] ← 语义相近，向量相近
「天气预报」 → [-0.45, 0.78, 0.23, ...] ← 语义无关，向量远离

度量方式：余弦相似度（Cosine Similarity），值域[-1, 1]，越接近1越相似。
RAG检索时：用户问题→嵌入→与知识库所有文档向量计算余弦相似度→返回Top-K最相似的文档。

稠密检索命中"住房公积金提取条件及流程"（语义匹配"取出来→提取"）
混合检索：两路结果通过RRF（Reciprocal Rank Fusion）融合，再用Cross-Encoder重排序 → 最终"住房公积金提取条件"排第一

Reranker是检索的"精排"环节——粗排召回Top-100，精排选Top-10。延迟+100ms，但召回质量+15-25%。不加Reranker的RAG等于裸跑
🔎 Anthropic Contextual Retrieval：检索准确率提升67%
传统RAG痛点：文档被切分为短文本块后，单个chunk丢失关键背景（如主体、时间、场景），导致"有相关信息但检索不到"。

解法：切分后用LLM为每个chunk生成一段简短的上下文描述（context），将context+chunk一起嵌入向量。每个chunk的向量既保留局部信息，又携带文档级上下文。
实测：检索失败率从5.7%降至1.9%，结合prompt caching，每百万tokens成本仅$1.02。

📚 CAG（Cache-Augmented Generation）：全量缓存，无需检索
核心思想：如果知识库不大（能塞进上下文窗口），就不做检索了——直接把全部知识缓存进KV-Cache，让模型一次看完再回答。

优势：无检索损失，信息完整，延迟最低（省去检索+重排步骤）
局限：知识量必须 < 上下文窗口（目前约10万-20万Token），超出则无法使用

类比：RAG是"查字典"（按需翻页），CAG是"背下来"（全量记忆）

与RAG的关系：不是替代，而是错位竞争。知识量小、查询频繁→CAG更优；知识量大、长尾查询多→RAG更优。
🚀 LLM Wiki（Karpathy, 2026.04）：知识编译器范式
核心范式转变：不是因为"用不用向量库"，而是"知识处理时机"的根本变化

传统RAG是解释器——每次查询临时检索碎片+拼接回答，查询结束知识不沉淀
LLM Wiki是编译器——资料入库时一次性结构化编译，产出永久Wiki页面

三层标准架构：
llm-wiki/
├── raw/        ← 第一层：原始文档（PDF/MD/网页），只读，Source of Truth
├── wiki/        ← 第二层：LLM编译产出的结构化Markdown页面，双向链接
└── schema.yaml ← 第三层：人类定义的入库规则、命名规范、巡检策略

工作流程闭环：Ingest（入库编译）→ Query（查询Wiki）→ Lint（健康巡检）→ 回到入库

入库时发生了什么：新资料丢进raw/ → LLM自动提取核心观点 → 关联检索已有词条 → 批量更新Wiki → 标注冲突
查询时怎么找相关页面：小规模全量加载，中大规模仍需全文/向量检索 → 这里会复用RAG技术栈
🔧 LLM Wiki 底层技术栈详解
入库（编译）阶段——极简方案：
- LLM：编译引擎，把raw文档转为wiki页面（入库可用强模型保证质量）
- 文件系统：存储raw/wiki/schema，纯Markdown文本
- Git：版本控制、溯源、回滚（强烈推荐）
不需要：向量数据库、Embedding模型、Reranker、BM25引擎

查询阶段——规模决定是否需要RAG：
- 小规模wiki（<10万字）→ 全量塞进上下文窗口 → 不需要RAG（本质是CAG）
- 中等规模wiki（10万-100万字）→ 需要检索wiki页面 → 可能需要RAG技术栈（对wiki页面做Embedding+向量检索，或ES全文搜索）
- 大规模wiki（>100万字）→ 几乎必然需要Embedding+向量检索wiki页面

关键洞察：RAG技术栈在这里换了检索目标——从"原始碎片chunk"变成"结构化wiki页面"。检索粒度更粗（whole page vs chunk），但质量更高（结构化 vs 碎片）。

2. 混合场景（核心知识+长尾临时查询）→ Wiki(核心80%) + RAG(长尾20%)。高频问题查Wiki，临时新增通知直接RAG检索原始文档

3. 纯LLM Wiki（极小规模）→ wiki规模<上下文窗口，全量加载，确实不需要RAG。但企业场景几乎不可能这么小

MetAPI — 中转站之上的元聚合

作者：cita-777 ｜ Stars：3.1k ｜ 协议：MIT ｜ 技术栈：Fastify + React 18 + TypeScript + Drizzle ORM

独特定位：中转站的中转站
不对接各模型厂商，而是对接各中转站本身——把分散的New API/One API/OneHub等站点聚合为统一入口。

解决的核心痛点：在多个中转站有账号时，每个站一个Key、余额分散、模型列表不同、某个站挂了要手动切换。
核心功能
自动模型发现：零配置发现上游所有站点的可用模型
智能路由引擎：按成本(40%)、余额(30%)、使用率(30%)加权分配，失败自动冷却+故障转移
四级成本信号：实测成本 → 账号配置成本 → 目录参考价 → 默认兜底
统一OpenAI + Claude下游格式，自动格式转换
自动签到：Cron定时执行各站签到领额度
集中看板：跨站余额分布、消费趋势、模型覆盖率对比
告警通知：Webhook/Bark/Server酱/Telegram/邮件 五渠道
模型操练场：在线交互测试
OAuth连接：Codex/Claude/Gemini CLI/Antigravity
局限
无多用户体系：单管理员架构，不适合对外分发Key
Key权限管理较简单：支持过期时间/费用上限/模型白名单/路由白名单，但无用户分组、兑换码、在线充值等
不直接对接模型厂商：必须先有上游中转站账号
适用场景
在多个中转站有账号、需统一管理和智能调度的重度用户。特别适合"薅羊毛"场景：多站签到领额度 + 智能路由选最便宜通道。
