课题0: ReAct循环原理实验

深入理解 Agent = LLM + Context + Tools 的核心运行机制，通过模拟器观察推理-行动-观察循环

步骤1: 理解Agent核心框架

Agent的核心公式非常简洁，却蕴含了智能体的全部奥秘：

核心公式：Agent = LLM + Context + Tools
三大组件缺一不可：
1) LLM（大语言模型）：Agent的「大脑」，负责推理和决策。没有LLM，Agent无法理解问题、无法规划行动、无法生成回答。
2) Context（上下文）：Agent的「工作记忆」，包括对话历史、系统提示、工具返回结果。没有Context，Agent每轮都是「失忆」状态，无法维持多轮对话。
3) Tools（工具）：Agent的「双手」，可调用的外部工具集（搜索、计算、数据库查询等）。没有Tools，Agent只能纯文本对话，无法与外部世界交互。

Agent = LLM + Context + Tools 不是一个简单的加法，而是一个涌现系统——三者协同产生的智能远大于各自之和。这就像人 = 大脑 + 记忆 + 双手，缺了任何一项都无法独立行动。
┌─────────────────────────────────────────────────┐ │ Agent 运行循环 │ │ │ │ ┌─────────┐ ┌──────────┐ ┌──────────┐ │ │ │ LLM │ ◄── │ Context │ ◄── │ Tools │ │ │ │ (推理) │ ──► │ (上下文) │ ──► │ (工具) │ │ │ └─────────┘ └──────────┘ └──────────┘ │ │ │ ▲ │ │ │ │ │ │ │ │ ▼ │ ▼ │ │ Thought ──► Action ──► Observation │ │ │ │ │ │ └──────► 更新Context ◄───────────────┘ │ │ │ │ ReAct循环：每轮 = Thought → Action → Observation │ │ 循环直到得到最终答案或达到最大迭代次数 │ └─────────────────────────────────────────────────┘

ReAct (Reasoning + Acting) 是目前最主流的Agent运行模式。它的核心思想是：Agent在每一步都先「思考(Thought)」再「行动(Action)」，然后「观察(Observation)」结果，再基于观察继续思考。这种「思考→行动→观察→再思考」的循环，让Agent能够处理复杂的多步推理问题。

关键洞察：为什么ReAct比纯推理更好？
纯推理（Chain-of-Thought）只靠LLM内部知识，遇到需要外部信息的问题就会产生幻觉（编造事实）。ReAct通过「行动」步骤让Agent可以查询外部知识、调用工具计算，从而获得真实的外部信息，大幅减少幻觉。

举例：用户问「2026年苏州公积金最新政策是什么？」
❌ 纯推理：LLM只能基于训练数据猜测，很可能编造过期政策
✅ ReAct：Agent搜索官方政策文档 → 获取真实信息 → 基于事实回答

步骤2: 运行完整ReAct循环

我们用一个政务场景来演示ReAct循环的完整过程。用户问的是「外地户口迁苏州，社保能一起转吗？」——这个问题需要综合查询户口迁移政策和社保转移政策，还要做数值判断（社保年限是否满足），恰好展示了ReAct多轮推理的优势。

1

迭代1：识别核心问题

Thought: 用户问的是户口迁移的条件，我需要先搜索苏州户口迁移政策
Action: 搜索("苏州户口迁移")
Observation: 苏州市户口迁移需满足：1)合法稳定住所；2)参加社保满3年；3)无犯罪记录。办理地点：辖区派出所户籍窗口。

2

迭代2：发现需要量化判断

Thought: 搜索结果提到需要社保满3年，我来算一下用户2018年开始交社保到现在够不够3年
Action: 计算器("2026 - 2018")
Observation: 计算结果: 8

3

迭代3：发现关联问题

Thought: 已交社保8年，满足3年条件。再搜索一下社保转移能否同步办理
Action: 搜索("苏州社保转移")
Observation: 苏州社保转移流程：1)原参保地开具参保凭证；2)新参保地提交转移申请；3)45个工作日内完成转移。线上可通过国家社会保险公共服务平台办理。

4

迭代4：信息充分，输出答案

Thought: 现在信息足够了，可以综合回答
Action: 完成("根据查询结果：1)户口迁移需合法稳定住所+社保满3年+无犯罪记录（您已交8年社保，满足条件）；2)社保转移可同步在线办理，通过国家社会保险公共服务平台，45个工作日完成。")

ReAct循环解析：为什么需要4轮迭代？
- 第1轮：识别核心问题（户口迁移）→ 调用搜索工具获取政策
- 第2轮：发现需要量化判断（社保年限是否满3年）→ 调用计算器工具
- 第3轮：发现关联问题（社保转移能否同步办）→ 再次搜索补充信息
- 第4轮：信息充分 → 综合输出最终答案

这种「思考→行动→观察→再思考」的循环就是ReAct的核心。注意第2轮和第3轮：Agent不是一次性搜索所有信息，而是在每轮观察后动态决定下一步需要什么。这就是ReAct与一次性检索的关键区别——ReAct是渐进式推理，每一步都基于前一步的观察结果。

步骤3: 上下文消融实验

消融实验（Ablation Study）是理解Agent组件重要性的关键方法：逐个移除上下文中的不同部分，观察Agent能力如何退化。这就像拆解一台机器看每个零件的作用。

消融实验原理：验证组件不可替代性
消融实验的逻辑非常简单：
1. 先用完整配置运行，得到基线性能（Baseline）
2. 逐个移除某个组件，观察性能变化
3. 性能下降越多 → 说明该组件越重要

在Agent上下文中，主要有以下可消融的组件：
- 对话历史：移除后Agent无法追问，多轮对话能力丧失
- 推理链（Thought链）：移除后复杂推理断裂，准确率下降
- 工具返回结果（Observation）：移除后等于无工具，回到纯聊天
- 全部移除只剩系统提示+问题：几乎无Agent能力
移除组件Token占比能力影响
完整上下文 (FULL)100%基线性能，完整能力
对话历史60%无法追问，多轮对话能力丧失
推理链75%复杂推理断裂，准确率下降
工具返回结果70%相当于无工具，回到纯聊天
仅系统提示+问题20%几乎无Agent能力
核心认知：Context是Agent的「工作记忆」
消融实验的核心结论是：Context中的每个组件都不可省略。就像人的工作记忆——如果你忘记了之前的对话内容、忘记了推理过程、忘记了查到的信息，你就不可能给出好的回答。Agent也是如此。

这也是为什么上下文管理（压缩、摘要、记忆系统）如此重要——我们既要保留关键信息，又要在有限的Token窗口内高效利用空间。

RAG核心流程：检索 → 增强 → 生成
当用户提问时，RAG的完整流程是：
1. 检索(Retrieval)：将用户问题向量化，在知识库中搜索语义最相近的文档片段
2. 增强(Augmented)：将检索到的文档片段作为上下文注入LLM的提示中
3. 生成(Generation)：LLM基于增强后的上下文生成回答

关键：temperature=0.3 确保回答确定性高（政务场景不能随意发挥），top_k=3 召回3条最相关文档（足够覆盖又不会引入太多噪声）。

工业场景与政务场景的核心差异：术语精确性要求高、安全警告必须突出、回答需结构化

创建工业运维知识库智能体

🎯
术语精确性

⚠️
安全警告

📋
结构化回答

构建工业知识库 + 问答测试

chunk_size=800 vs 500：为什么工业文档需要更大的分块？
政务FAQ每条仅几十字，chunk_size=500足够。
但工业维修文档的步骤通常是连续的（如「断电→拆外壳→检查接线→测量电压→更换元件→测试→装回」），如果分块太小会把一个完整维修步骤截断，导致检索到的只是半截信息。

课题2.5: RAG原理深入实验

从嵌入向量到检索策略到Agentic RAG，深入理解RAG的每一个环节

步骤1: 稠密嵌入原理 — 语义相近→向量相近

嵌入(Embedding)是RAG的基石。它将文本转化为高维向量，使语义相近的文本在向量空间中距离更近。

文本 ──[嵌入模型]──► 向量（如768维浮点数组） 例： 「公积金提取」 → [0.12, -0.34, 0.56, ...] ← 语义A 「住房公积金取出」 → [0.11, -0.32, 0.55, ...] ← 语义A（相近！） 「天气预报」 → [-0.45, 0.78, 0.23, ...] ← 语义B（远离！） 余弦相似度： cos(A, A') ≈ 0.98 ← 语义相近，向量相近 cos(A, B) ≈ -0.30 ← 语义无关，向量远离
嵌入原理详解：为什么语义相近的文本向量也相近？
嵌入模型（如BGE-M3、text-embedding-ada-002）在海量文本上训练，学习到：
- 语义相似的句子 → 映射到向量空间中相近的位置
- 语义不同的句子 → 映射到向量空间中远离的位置

核心公式是余弦相似度：cos(A,B) = (A·B) / (|A|×|B|)
值域[-1, 1]：1表示完全相同，0表示无关，-1表示相反

这就是RAG检索的基础：用户问题向量 → 与知识库中所有文档向量计算余弦相似度 → 取top_k个最相似的文档
Python
class EmbeddingVisualizer: """嵌入向量可视化与对比工具""" @staticmethod def cosine_similarity(v1: List[float], v2: List[float]) -> float: """计算余弦相似度""" dot = sum(a * b for a, b in zip(v1, v2)) norm1 = sum(a * a for a in v1) ** 0.5 norm2 = sum(b * b for b in v2) ** 0.5 return dot / (norm1 * norm2) if norm1 * norm2 > 0 else 0 @staticmethod def mock_embed(text: str, dim: int = 8) -> List[float]: """模拟文本嵌入（用哈希生成确定性向量，仅用于教学演示）""" h = hashlib.md5(text.encode()).hexdigest() return [((int(h[i:i+2], 16) / 255) - 0.5) * 2 for i in range(0, min(dim * 2, len(h)), 2)] def demo_semantic_similarity(self): sentences = [ "公积金怎么提取", "住房公积金如何取出", "社保转移怎么办", "今天的天气怎么样", ] embeddings = {s: self.mock_embed(s) for s in sentences} for i, s1 in enumerate(sentences): for j, s2 in enumerate(sentences): if i < j: sim = self.cosine_similarity(embeddings[s1], embeddings[s2]) print(f" 「{s1[:8]}」 vs 「{s2[:8]}」 余弦相似度: {sim:+.3f}")

步骤2: 三种检索策略对比

检索方式原理优势劣势
稀疏检索(BM25)关键词TF-IDF匹配精确术语匹配强无法理解同义词
稠密检索语义向量相似度模糊语义匹配强精确术语可能漏
混合检索两路融合+重排序兼顾精确和模糊计算成本更高
混合检索 = 稠密(语义) + 稀疏(关键词) + 重排序
混合检索通过 Reciprocal Rank Fusion (RRF) 融合两路结果：
- 稠密检索找到语义相近的文档（如「公积金怎么取」匹配到「住房公积金提取条件」）
- 稀疏检索找到关键词匹配的文档（如「公积金」精确匹配）
- RRF将两路排名融合，取综合排名最优的文档
- 再用 Cross-Encoder 对融合结果重排序，提升精度

实际部署建议：优先使用混合检索，兼顾语义和精确匹配。

步骤3: 被动RAG vs Agentic RAG

传统被动RAG只做1次检索，不管结果够不够；Agentic RAG让Agent自主决定是否检索、是否需要再次检索。

╔══════════════════════════════════════════════════════════╗ ║ 传统被动 RAG 流程 ║ ║ ║ ║ 用户问题 ──► 检索1次 ──► 拼接上下文 ──► LLM生成回答 ║ ║ ║ ║ 特点：检索是「被动」的，只做1次，不管检索结果够不够 ║ ╚══════════════════════════════════════════════════════════╝ ╔══════════════════════════════════════════════════════════╗ ║ Agentic RAG 流程 ║ ║ ║ ║ 用户问题 ──► Agent思考 ──► 需要检索吗？ ║ ║ │ │ ║ ║ │ ┌────┴────┐ ║ ║ │ │不需要 │需要 ║ ║ │ ▼ ▼ ║ ║ │ 直接回答 检索+评估 ║ ║ │ │ ║ ║ │ ┌─────┴──────┐ ║ ║ │ │结果够吗？ │ ║ ║ │ 够了│ │不够 ║ ║ │ ▼ ▼ ║ ║ │ 生成回答 换关键词再检索 ║ ║ │ │ ║ ║ │ └──► 回到评估(最多N次) ║ ╚══════════════════════════════════════════════════════════╝
场景被动RAGAgentic RAG
简单：「新生儿落户要什么材料？」检索1次→命中→回答 ✅思考→检索→评估(够了)→回答 ✅
复杂：「外地户口迁苏州，社保转移能一起办吗，要多久？」检索1次→命中户口但漏社保时限→回答不完整 ❌检索户口→评估(缺社保)→检索社保→完整回答 ✅
超范围：「苏州哪里有好吃的？」检索1次→无命中→仍尝试生成→可能幻觉 ⚠️思考→决定不检索→直接告知超出范围 ✅
Agentic RAG 三大核心能力
1. 自主决策：判断是否需要检索（简单问题可以直接回答，节省检索开销）
2. 自主迭代：检索结果不够时自动换关键词再检索（多轮检索确保完整性）
3. 自主拒绝：超出知识范围时直接说「不知道」，避免幻觉

本质区别：被动RAG是「检索管道」，Agentic RAG是「有判断力的检索员」。TeleAgent平台默认使用Agentic RAG模式。

课题3.5: 上下文压缩策略实验

长对话场景下Token不够用怎么办？6种压缩策略帮你解决

步骤1: 为什么需要上下文压缩？

上下文压缩的必要性
LLM上下文窗口有限（4K/8K/128K tokens），长对话场景下历史消息会逐渐占满窗口。问题不仅是「装不下」，还有：
- 延迟：Token越多，推理延迟越高
- 成本：按Token计费，Token越多费用越高
- 噪声：过多无关历史干扰LLM判断

压缩策略的核心目标：在保留关键信息的前提下减少Token消耗

6种压缩策略概览

策略原理适用场景
none不压缩，保留全部历史短对话
sliding_window保留最近N轮简单场景，最常用
summary早期对话压缩为摘要需要上下文但不需原文
key_facts提取关键信息信息密度高的场景
summary_with_reference摘要+引用编号需要回溯原文
hierarchical分层级：最近完整+中期摘要+早期关键超长对话
