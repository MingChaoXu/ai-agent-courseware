政务服务全链路智能服务

5个专用Agent协同 + System Hint机制 + 事件触发 · 50分钟

核心目标

构建一个覆盖「问答→推荐→办理→校验→审核」全链路的政务服务智能体系统，掌握多Agent协同编排、System Hint动态注入、事件触发自动启动工作流三大核心能力。

⚡ 核心架构原理：多Agent协同 = 专业化分工 + 结构化协作
单Agent处理复杂业务时，system_prompt会膨胀到难以维护，且不同职责相互干扰。多Agent协同的核心思想：每个Agent只做一件事，做到极致，再通过工作流串联。这就像政务大厅的窗口分工——咨询台、导办台、受理窗口、审核窗口各司其职。

关键技术点：
1. system_prompt精炼化：每个Agent的职责描述控制在5-8行，包含角色定义 + 职责范围 + 输出格式三元组
2. System Hint注入：运行时动态插入时间戳、工具调用计数、TODO列表等上下文，让Agent"感知"业务状态
3. 事件驱动触发：用户提交申请→事件总线→自动启动全链路工作流，无需人工调度

🔧 Agent设计原则：单职责 + 输出标准化
每个Agent的system_prompt遵循固定结构：
1. 角色声明：「你是XXAgent」——明确身份
2. 职责定义：「职责：XXX」——划清边界
3. 处理规则：列出3-5条具体规则
4. 输出格式：指定JSON/结构化输出，确保下游Agent可解析

为什么要标准化输出？因为工作流中Agent的输出是下一个Agent的输入。如果问答Agent输出的是自然语言，推荐Agent就无法精准提取业务类型。因此，问答Agent输出{"intent":"公积金业务","keywords":["提取","购房"],"urgency":"普通"}，下游Agent直接消费。

1.2 System Hint机制

💡 核心洞察：System Hint = 连接Agent与业务状态的桥梁
传统Agent不知道"现在几点"、"还有什么没做"、"刚才调用了几次工具"。System Hint通过在system_prompt末尾动态注入上下文块，让Agent获得业务感知能力。

[System Hint]
当前时间: 2026-07-08 14:30:00
当前智能体: 政务问答Agent
已调用工具次数: 0
待办事项: 无
额外信息: 需判断6类业务类型

这个机制的精妙之处：每次事件触发后，System Hint自动更新。当用户提交申请时，TODO列表自动增加"校验材料完整性"等条目，Agent就能在后续对话中主动提醒，而非被动等待。
1

构建System Hint函数

_build_system_hint(agent_name, extra)：接受Agent名称和额外信息，生成包含时间戳、工具计数、TODO列表的Hint块。每次Agent被调用前动态注入，确保信息时效性。

2

注入到system_prompt末尾

将Hint块拼接到system_prompt字符串末尾：基础Prompt + qa_hint。LLM在推理时会优先关注Prompt末尾的内容，因此Hint放在末尾效果最佳。

3

事件触发后自动更新

通过EventDrivenAgent.update_system_hints()方法，在每次事件处理后自动刷新时间戳和TODO状态，确保下一次Agent调用获取最新上下文。

1.3 事件触发机制与5步工作流

⚡ 架构演进：从"手动调用"到"事件驱动"
传统方式：人工触发每个Agent——手动调问答、手动传结果给推荐、手动填表单...

社会治理综合智能体

4个Agent协同 + 异步处理(threading) + 条件分支 · 40分钟

核心目标

构建综治中心的多Agent协同系统，掌握异步并行处理（事件录入和态势预警同时执行）和条件分支（仅"矛盾纠纷"类型触发法律Agent）两个高级工作流模式。

⚡ 核心架构原理：异步+条件 = 高效+精准
异步处理：当两个Agent之间没有数据依赖时，可以并行执行。例如"事件录入"和"态势预警"——录入是处理当前事件，预警是分析月度趋势，两者互不依赖，并行可节省50%时间。

条件分支：不是每个事件都需要走所有Agent。矛盾纠纷类事件才需要法律咨询Agent，安全隐患类事件则直接进入简报生成。这通过工作流step中的condition字段实现："condition": "${standard_event.type} == '矛盾纠纷'"

异步处理架构

社会治理工作流（异步+条件分支）： 事件报告 ──► [Step1] 事件信息Agent（标准化录入） │ ┌─────────┴──────────┐ │ │ [Step2]法律咨询Agent [Step3]态势预警Agent 条件: type=="矛盾纠纷" 与Step2并行执行 │ │ └─────────┬──────────┘ │ [Step4] 简报生成Agent（汇总所有结果）

Python threading实现：使用threading.Thread和threading.Barrier协调两个并行线程。Barrier确保两个线程都完成后再进入简报汇总阶段。

2.2 条件分支详解

🔧 条件分支实现：工作流Step的condition字段
在工作流定义中，每个step可以添加condition字段，值为一个表达式。运行时，引擎会评估该表达式，仅当结果为True时执行该步骤。

示例："condition": "${standard_event.type} == '矛盾纠纷'"

这意味着：
- 上报漏水纠纷 → 事件Agent输出type="矛盾纠纷" → 法律Agent执行 → 简报包含法律建议
- 上报安全隐患 → 事件Agent输出type="安全隐患" → 法律Agent跳过 → 简报不含法律建议

条件分支让工作流"智能路由"，不同类型的事件走不同处理路径，避免无关Agent的无效调用。

事件驱动Agent实战

三种触发模式 + System Hint集成 + 被动vs事件驱动对比 · 30分钟

核心目标

掌握EventDrivenAgent架构的搭建与三种触发模式的实现，理解事件驱动如何让Agent从"工具"升级为"协作伙伴"。

⚡ 核心架构原理：EventDrivenAgent = 事件源 + 事件总线 + 处理器 + 智能体推理
传统Agent架构：用户提问 → Agent回答（被动模式）
事件驱动架构：事件源 → 事件总线 → 处理器 → 智能体推理 → 响应（主动模式）

三种触发模式覆盖90%的运营自动化场景：
1. 定时触发：cron调度器 → 每天9:00商机跟进提醒
2. 数据触发：监控阈值 → 缺陷率超3%自动告警
3. 外部触发：API回调 → 客户提交申请启动审核

核心代码模式：eda.register_handler(event_type, handler) + eda.emit_event(event)

3.1 三种触发模式详解

1

定时触发：每天9:00商机跟进提醒

场景：客户经理每天早上需要跟进待办商机，但容易遗漏。通过timer事件类型，cron调度器在9:00自动发射事件，Agent生成跟进提醒并推送给对应客户经理。

事件数据：{"task": "商机跟进提醒", "target": "苏州XX电子有限公司", "scheduled_time": "09:00", "last_follow_up": "3天前"}

2

数据触发：缺陷率超阈值告警

场景：制造业生产线缺陷率超过3%时需要立即响应。监控系统实时检测，当PCB板缺陷率 = 5.2%超过阈值，自动发射data_alert事件，触发根因分析和告警通知。

事件数据：{"metric": "PCB板缺陷率", "value": "5.2%", "threshold": "3.0%", "production_line": "A3线", "trend": "连续3天上升"}

3

外部触发：客户提交申请→启动审核

场景：客户在自助平台提交专线扩容申请，系统通过API回调发射external_application事件，自动启动审核工作流，无需人工转交。

事件数据：{"applicant": "苏州XX科技有限公司", "business_type": "专线扩容", "amount": "50万"}

3.2 System Hint集成

💡 System Hint的三层信息注入
事件驱动Agent的System Hint包含三层信息：

第一层：时间感知
当前时间: 2026-07-08 14:30
让Agent知道"现在几点"，可以判断"3天前的跟进需要催办了"

第二层：TODO感知
待办事项数: 3
待办清单: [high] 跟进苏州XX公司商机, [high] 处理缺陷率告警, [medium] 审核专线扩容申请
让Agent知道"还有什么没做"，可以主动提醒

第三层：状态感知
已注册事件处理器: 3
让Agent知道"当前系统能处理哪些事件"

这三层信息通过eda.update_system_hints()自动更新，然后拼接到Agent的system_prompt末尾，实现业务状态与Agent推理的无缝连接。

3.3 被动模式 vs 事件驱动模式 全面对比

维度被动模式事件驱动模式
触发方式用户主动提问事件自动触发
时间感知无，不知道当前时间有，System Hint注入时间戳
任务感知无，不知道待办事项有，TODO列表自动注入
主动性被动等待主动提醒+自动执行
适用场景简单问答运营巡检+商机跟进+风险告警
开发复杂度低中（需事件总线+处理器注册）
典型代码量~20行~80行（含事件注册+处理器）
Agent角色工具（被动回答）协作伙伴（主动服务）
💡 核心洞察：事件驱动让Agent从「工具」升级为「协作伙伴」
1. 被动模式的Agent只是一个"高级搜索引擎"——你问它答，不问不动。
2. 事件驱动模式的Agent是一个"智能助手"——它会主动提醒你跟进商机、自动检测风险、在客户提交申请时立即启动流程。
3. System Hint是连接Agent与业务状态的桥梁——让Agent"感知"时间、TODO、系统状态。
4. 三种触发模式（定时/数据/外部）覆盖了90%的运营自动化场景。

智能体消融评估实战

消融实验 + 量化分析 + 评估报告 · 50分钟

核心目标

掌握消融评估方法——通过系统化移除Agent组件来量化各部分贡献度，回答"到底哪个组件最重要"这个关键问题。

⚡ 核心架构原理：消融实验 = 科学验证组件价值
消融实验（Ablation Study）源自神经科学研究：移除大脑某区域，观察功能变化，从而推断该区域的功能。

在Agent评估中，消融实验的方法：
1. 建立基线（baseline）：完整配置的Agent
2. 逐个移除组件：移除system_prompt、移除工具描述、改非正式语气...
3. 在相同测试集上评估每个变体
4. 计算各组件的贡献度 = |变体得分 - 基线得分| / 基线得分 × 100%

关键公式：contribution_pct = |delta| / baseline_score * 100%
影响等级：🔴 重大 >30% 🟡 显著 >15% 🟢 轻微 ≤15%

4.1 建立评估基准

5个测试问题

  1. 公积金购房提取需要哪些材料？
  2. 户口从外地迁入苏州怎么办理？
  3. 社保断缴3个月会有什么影响？
  4. 不动产登记需要多长时间？
  5. 婚姻登记可以跨区办理吗？

4维度评估函数

🔧 evaluate_answer(question, answer) → 0.0~1.0
维度1：完整性（0.3分）
回答长度 > 50字 → +0.2，> 150字 → +0.1
衡量Agent是否给出了充分的信息

维度2：准确性（0.3分）
根据问题关键词匹配答案中的专业术语。例如"公积金"问题检查["提取","账户","余额","购房"]的命中率
命中率 = 匹配词数 / 总关键词数 × 0.3

维度3：结构性（0.2分）
检查回答中是否有结构化标记：编号(1. 2.)、步骤(① ②)、列表(- )
有任意一个 → +0.2

维度4：政策引用（0.2分）
检查回答中是否有政策引用标记：规定、政策、条例、办法、通知
有任意一个 → +0.2
